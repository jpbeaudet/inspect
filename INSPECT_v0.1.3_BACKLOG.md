# Inspect CLI — v0.1.3 Patch Backlog

**Rule:** Ship when 7+ items accumulated OR one critical issue found.
**Source:** Known limitations from pre-v0.2.0 roadmap + retrospective items surfaced during v0.1.2 bundle implementation + field feedback from **four independent v0.1.2 users** (Keycloak / Vault debug sessions on multi-container hosts; the 3rd user's BUG-3 deploy contributed both the cache-freshness ask now tracked as F8 and the silent-stdin-drop bug now tracked as F9; the 4th user's ~15-min real-ops session contributed the polish bundle now tracked as F10 — selector-error hints, `inspect cat --lines`, `why` non-service chained hint, `grep` verb docstring, plus an independent confirmation of F7.4 `--quiet`) **+ a multi-hour destructive migration session by the primary operator** (60-image atlas-centralization migration driven entirely through `inspect run` from a Codespace) which contributed F14 (script mode), F15 (file transfer), F16 (streaming), F17 (multi-step exit codes), F18 (session transcript), and refinements to F10 / F11 / F13.
**Status:** **OPEN — committed to ship all 25 items.** This is the final scope; v0.1.3 freezes after this list and implementation begins. v0.1.3 is the last "break things freely" release on the docker-only surface; **v0.1.4 is dedicated to Kubernetes** (medium = k8s, kubectl-equivalent verbs, k8s-aware selectors). The stabilization sweep that was previously planned for v0.1.4 shifts to a v0.1.5 stabilization release before the v0.2.0 contract. Because v0.1.4 will be heads-down on k8s, **anything docker / compose / SSH related that doesn't ship in v0.1.3 will not be touched again until v0.1.5 at the earliest** — that's the reason to clear the entire v0.1.3 backlog rather than slipping items.
**Contract:** No backward compatibility. Break whatever needs breaking. After v0.2.0 the CLI surface, JSON schema, and config formats are frozen.

### F2 — `docker inspect` batched-timeout warning noise during setup
**Source:** v0.1.2 field feedback (independently re-reported by 2nd field user — "appeared as a warning every single `setup` run" on a 37-container Keycloak host with the 10s default).
**Severity:** Low (cosmetic, but it is the *first* output a new user sees on `inspect setup` / first discovery — and a scary warning on first run erodes trust before the tool has done anything useful). **Problem:** During setup / first discovery, the batched `docker inspect` probe emits a timeout warning that (a) appears even on healthy hosts, (b) does not actually indicate a failure (discovery succeeds, services are populated, the warning is just noise), and (c) uses a fixed 10s default that is too aggressive for hosts with 30+ containers. Today's behavior trains users to ignore warnings, which is exactly the wrong reflex.
**Fix:**
- **Decide what the warning means** before changing anything. Audit `src/discovery/probes.rs` (the docker-inspect batch path) and classify the timeout into one of three buckets:

  2. **Slow but successful** — batch took longer than the soft threshold but returned complete data. **Demote to debug-level only**, visible under `--debug` / `RUST_LOG=debug`. Default output stays clean.
  3. **Per-container timeout in a parallel batch** — surface once at the end of discovery as `warning: docker inspect timed out for N/M containers; rerun with --force or check daemon load` (single line, with counts), never per-container.

### F4 — `inspect why` compose-aware deep-diagnostic bundle
**Source:** v0.1.2 field feedback (2nd field user, "the one load-bearing feature request"). **Problem:** `inspect why <selector>` reports up/down + a single likely-root-cause line + a `NEXT:` hint. 2. **Effective `Cmd` + `Entrypoint`** — the `Cmd` and `Entrypoint` fields from `docker inspect`, **plus** the resolved entrypoint script if it is a known `docker-entrypoint.sh`-style wrapper (read the first 50 lines of the script to surface flag injection like Vault's `-dev-listen-address`). ### F5 — Container-name vs compose-service-name uniform resolution
**Source:** v0.1.2 field feedback (2nd field user). The operator tried `arte/luminary-onyx-onyx-vault-1` (the docker container name from `docker ps`), got an error, then `arte/onyx-vault` (the compose service name) worked. Both forms appear in the discovered inventory, so the failure is surprising.
**Severity:** Small/medium (every compose user hits this once; trust cost on first encounter).
**Problem:** A single container has at least two valid identifiers in the discovered inventory: the docker-assigned container name (`<project>-<service>-<index>`, e.g. `luminary-onyx-onyx-vault-1`) and the compose service name (`onyx-vault`). The selector resolver currently accepts only one of them depending on the verb / code path, and the error on the rejected form is generic ("no targets" or "invalid selector") with no hint at the canonical form.
**Fix:**
- **Single resolution pass** in `src/selector/`: every container selector resolves through one function that tries, in order: (1) exact match against compose `service` name, (2) exact match against docker container name, (3) glob match against either. The function returns the canonical form (compose service name, when available) plus a list of aliases.
- All verbs (`why`, `logs`, `run`, `ports`, `cat`, `exec`, `health`) use that function. No verb-local resolution paths.
- When the rejected form is a known docker container name but the canonical is the compose service: error becomes `error: 'arte/luminary-onyx-onyx-vault-1' is the docker container name; the canonical selector is 'arte/onyx-vault' (try that, or use the docker name with --by-container)`.
- New flag `--by-container` (and config option `selector.prefer = "container" | "service"`, default `"service"`) for operators who explicitly want docker-container-name semantics (e.g. when two compose services map to the same image and they want the specific instance).
- `inspect status <ns>` and `inspect health <selector>` JSON output gains an `aliases: ["luminary-onyx-onyx-vault-1"]` field per service so agents discover the equivalence without trial-and-error.

**Test:**
- `inspect why arte/onyx-vault` and `inspect why arte/luminary-onyx-onyx-vault-1` both succeed and target the same container.
- Without the disambiguation flag, the docker-container form prints the canonical-form hint as part of stderr but still resolves (warning, not error). With `selector.prefer = "service"` (default), this warning fires; with `selector.prefer = "container"`, it does not.
- Two compose services running the same image: `--by-container` resolves to the specific docker instance; default service-name resolution returns the compose service.
- `inspect status arte --json` output includes the `aliases` field for every service that has a docker-container-name distinct from its compose-service name.
- Glob form: `arte/onyx-*` and `arte/luminary-onyx-onyx-*-1` both work; the result set is identical when each compose service has exactly one container.

### F6 — First-class `inspect compose` verbs
**Source:** v0.1.2 field feedback (2nd field user, "no obvious `inspect compose` integration … first-class compose verbs would replace 80% of my `run` usage"). Field operator was repeatedly running `inspect run arte -- 'cd /opt/luminary-onyx && sudo docker compose …'` for ps / logs / restart / config. Compose is the dominant deployment shape on the field servers Inspect targets today.
**Severity:** Medium-large (new verb surface, but each sub-verb is a thin wrapper around an existing remote `docker compose` invocation — risk is in scoping, not in implementation complexity).
**Problem:** Compose projects are a first-class deployment unit on the hosts Inspect targets, but Inspect treats them as opaque collections of containers. To inspect a compose project's effective config, restart a single service, or view aggregated compose logs, operators drop back to `inspect run <ns> -- 'cd <project_dir> && sudo docker compose …'` — losing structured output, audit trail, redaction, and selector grammar.
**Fix:** Ship a small, **read-mostly** `inspect compose` subcommand surface in v0.1.3. Write verbs are limited to `restart` (the safest, most-needed action). `up` / `down` / `pull` are deferred to **v0.1.5+** pending a compose-write design review (v0.1.4 is the k8s release and will not touch compose).

Sub-verbs in v0.1.3:
- `inspect compose ls <ns>` — list compose projects discovered on the namespace, with `name`, `working_dir`, `service_count`, `running_count`, `compose_file`. Replaces `docker compose ls`.
- `inspect compose ps <ns>/<project>` — per-service status table for one project (state, ports, image, uptime). Replaces `docker compose ps`.
- `inspect compose config <ns>/<project>` — effective merged compose config (resolved variables, profiles applied). Replaces `docker compose config`. Streamed through redaction.
- `inspect compose logs <ns>/<project>[/<service>]` — aggregated logs for a project (or one service inside it). Wraps `docker compose logs` with the existing `--tail` / `--follow` / `--since` flags from `inspect logs`.
- `inspect compose restart <ns>/<project>/<service>` — restart a single service. **Audited** like any write verb (`verb=compose.restart`), respects `--dry-run`. Aborts if more than one service is targeted unless `--all` is passed (defensive default).

Design points:
- Compose project discovery extends `src/discovery/probes.rs` to find compose projects via `docker compose ls --format json`. Cached alongside the existing container inventory; surfaces in `inspect status <ns>` as a new `compose_projects:` line.
- Project paths resolve against the discovered `working_dir`; operators never type the path. Selector form is `<ns>/<project>` for the project and `<ns>/<project>/<service>` for a service inside it. The existing `<ns>/<service>` form continues to work because F5's resolver tries compose-service first.
- All sub-verbs share the existing `SUMMARY` / `DATA` / `NEXT` output discipline and `--json` schema.
- **Out of scope for v0.1.3:** `up`, `down`, `pull`, `build`, `exec`. Those introduce compose-state-mutation semantics that need their own design pass; deferred to the **v0.1.5 backlog** with a placeholder (v0.1.4 is k8s-only).
- Audit log: every `compose restart` writes a structured entry with `project`, `service`, `compose_file_hash` (so the post-mortem can verify the file didn't change between audit and rerun).
- Help: `inspect help compose` lists the sub-verbs and explicitly notes which compose actions are intentionally not yet exposed and why.

**Test:**
- Mock medium with two compose projects: `inspect compose ls arte` lists both with correct service counts.
- `inspect compose ps arte/luminary-onyx` returns the per-service table; `--json` schema matches the spec.
- `inspect compose config arte/luminary-onyx` produces the merged YAML and runs through the redaction pipeline (L7-aware once that lands; today the existing env-var masker applies).
- `inspect compose logs arte/luminary-onyx` aggregates; `inspect compose logs arte/luminary-onyx/onyx-vault` narrows to one service.
- `inspect compose restart arte/luminary-onyx/onyx-vault` writes an audit entry, restarts only that service, and exits 0.
- `inspect compose restart arte/luminary-onyx` (no service) exits 2 with "specify a service or pass --all".
- `inspect compose restart arte/luminary-onyx --all --dry-run` lists every service that *would* restart without doing it.
- `inspect status arte` output includes `compose_projects: 2` (or whatever the count is); `--json` includes the structured project list.
- `inspect compose up`, `inspect compose down`, `inspect compose pull` exit 2 with "intentionally not implemented in v0.1.3 — see `inspect help compose`".

### F7 — Selector / output ergonomic papercuts (7 small fixes bundled)
**Source:** v0.1.2 field feedback (2nd field user "minor papercuts" + 3rd field user "flaky / surprising" sections). Bundled because each fix is < 30 LOC and they all touch error formatting / selector parsing / output trimming / phrasing.
**Severity:** Low individually; collectively they remove seven obvious "the tool should have just told me" moments from a fresh user's first hour.
**Problem + Fix (one bullet each):**

1. **Pre-setup verb error points at `inspect profile` instead of `inspect setup <ns>`.** Today, `inspect logs arte/onyx-vault` before discovery has run errors with `servers tried: arte / services available: (none)` and a hint pointing at `inspect profile`. The operator wanted "run `inspect setup arte` first". Fix: detect the "namespace known, services empty" case in `src/commands/logs.rs` (and every other read verb) and emit the chained hint `→ run 'inspect setup <ns>' to discover services on this namespace`. The `inspect profile` hint stays for the genuinely-misconfigured-namespace case (namespace not in `servers.toml`).

2. **`arte:` shorthand for host-file paths is rejected with a generic "invalid selector character ':'" error.** The correct form is `arte/_:/path` (the `_` indicates "host, not container"). Fix: in the selector parser (`src/selector/parse.rs`), specifically detect the `<ns>:<absolute_path>` shape and emit `error: 'arte:/path' looks like a host-path selector — did you mean 'arte/_:/path'? (the '_' selector targets the host filesystem)`. Pure error-message change, no parser-grammar change.

3. **`inspect ports arte | grep 8200` ergonomics.** Operators want a structured filter, not a grep. Fix: add `--port <n>` (single port) and `--port-range <lo-hi>` to `inspect ports`, filtering the table server-side. `inspect ports arte --port 8200` returns only rows where host or container port matches 8200. JSON output respects the same filter.

4. **`inspect logs --tail` collides with the user's own `| tail` because the `SUMMARY` block is at the end.** Fix: add a global `--quiet` flag that suppresses the trailing `SUMMARY` and `NEXT:` blocks (keeps `DATA` only), making output safe to pipe into `tail`, `head`, `grep -A`, etc. without worrying about trailer corruption. Documented in `docs/MANUAL.md` under "piping output". `--quiet` is mutually exclusive with `--json` (json output is already trailer-free).

5. **`inspect status` empty-state reads as "everything is broken."** (3rd field user.) Today's output `0 service(s): 0 healthy, 0 unhealthy, 0 unknown` is alarming when the actual meaning is "no service definitions matched my heuristics" — the host can be fully up and healthy. Fix: when the discovered service set is empty *and* the underlying inventory is non-empty (i.e. containers exist but none were classified as services), replace the zeros line with `no service definitions configured for <ns> — N container(s) discovered but unmatched. Run 'inspect ps <ns>' to see them, or 'inspect setup <ns> --force' to re-classify.` When the inventory is genuinely empty, current zeros line stays (it is correct then). The `--json` output gains an explicit `state: "no_services_matched" | "empty_inventory" | "ok"` field so agents distinguish the three cases without parsing prose.

6. **`inspect connect` prints `Permission denied (publickey)` *before* prompting for the passphrase.** (3rd field user.) Functionally fine — the second auth attempt with the unlocked key succeeds — but a scanner-style agent reads the first line as a hard failure and bails. Fix: in `src/commands/connect.rs`, when the first publickey attempt fails because the key is encrypted, **suppress the SSH client's `Permission denied` line** (filter on the connect path only, not in `inspect run` / `exec` output) and print our own line: `key requires passphrase — prompting…` immediately before the prompt. The actual `ssh` debug output stays available under `--debug`. This is a stderr-line-filtering change, not a protocol change; the `ssh -v` log on `--debug` is unchanged so genuine auth failures remain diagnosable.

7. **`inspect status` is the last verb without `--json`.** (3rd field user honourable mention: "the gap is `status`.") F1 already touches the status code path; while there, add a `--json` output mode that emits the same fields shown in the text table plus the `state` field from papercut #5 above. Schema goes into `tests/help_json_snapshot.rs`. This is the last gap in the "JSON everywhere" story before v0.2.0 freezes the schema.

**Test:**
- Pre-setup: `inspect logs arte/onyx-vault` before any discovery for `arte` exits 2 with the chained hint pointing at `inspect setup arte`. Same for `inspect run`, `inspect why`, `inspect cat`.
- `inspect cat arte:/etc/hosts` exits 2 with the `arte/_:/etc/hosts` suggestion. `inspect cat arte/_:/etc/hosts` succeeds.
- `inspect ports arte --port 8200` returns only matching rows; `--port-range 8000-9000` returns the range; `--port 8200 --json` schema is unchanged minus the filter.
- `inspect logs arte/onyx-vault --tail 50 --quiet | tail -10` prints exactly 10 log lines with no `SUMMARY` / `NEXT` text appended.
- `inspect status arte --json` schema matches the snapshot in `tests/help_json_snapshot.rs` and includes `services`, `counts`, and `state`.

### F8 — Cache freshness: `--refresh` flag, `SOURCE:` provenance line, tiered TTL on read verbs
**Source:** v0.1.2 field feedback (3rd field user, BUG-3 deploy — explicitly tagged "the one load-bearing feature request" and "if you can only ship one thing in v0.1.3, ship that"). Real-world session: operator ran `inspect restart arte/atlas-vault --apply`, the restart succeeded and the fleet became healthy, but `inspect why arte/atlas-vault` and `inspect status arte` continued to report the pre-restart unhealthy state served from the post-`setup` cache. The operator nearly concluded the fix had not taken effect; only `inspect run arte 'docker ps …'` confirmed reality. The wrong `NEXT:` suggestion problem (3rd user item #5 — `why` keeps suggesting `restart` after a successful restart) is the same root cause: stale runtime state in the discovery cache.
**Severity:** **High value, medium effort.** Without this, every diagnostic verb's output is silently a snapshot of unknown age, and the entire `why` / `status` / `health` / `ports` tier becomes untrustworthy after any mutation. Operators learn to distrust the tool and fall back to `ssh`, which is the exact failure mode this CLI exists to prevent. F1's fix and F4's deep-bundle both depend on this — fresh data flowing into a more detailed diagnostic is only useful if the operator can tell it is fresh.
**Problem:** The post-`setup` profile is treated as ground truth indefinitely by every read verb. There is no flag to bypass it, no indication in output that the answer came from cache vs. live, and `inspect setup --force` is the only way to refresh — which is heavy (full re-discovery of 30+ containers) and not a sensible per-diagnostic action. The result: a freshly-restarted service reads as "still unhealthy" until the next `setup`, and the audited mutation surface (the strongest part of the tool) is undermined by the read surface lying about its effects.
**Fix:** Make the cache visible, tiered, and per-verb refreshable.

1. **Tiered cache** (extends `src/discovery/engine.rs`):
   - **Inventory tier** — slow-changing facts: container existence, names, images, declared ports, compose project membership. Default TTL **5 minutes**. Survives across `inspect` invocations.
   - **Runtime tier** — fast-changing facts: running/exited, health status, exit code, restart count, port-binding reality, recent log tail. Default TTL **10 seconds**. Re-fetched on most read verbs.
   - The two tiers are stored in separate cache files (`~/.inspect/cache/<ns>/inventory.json` + `~/.inspect/cache/<ns>/runtime.json`) with separate mtimes so freshness is computed per tier.
   - Verbs declare which tier(s) they need. `status`, `why`, `health`, `ports` need both. `ps` needs runtime only. `images`, `volumes`, `network` need inventory only.
2. **Top-level `--refresh` (alias `--live`) flag** on every read verb that consults the cache: `status`, `why`, `health`, `ps`, `volumes`, `images`, `network`, `ports`, `logs --tail` (when reading the cached service list to resolve the selector). Behavior: invalidate and rewrite the runtime tier (and, if the verb needs it, the inventory tier) before serving. Cheap for runtime tier (one batched `docker ps` + `docker inspect`); expensive only when the operator opts in via `--refresh-inventory` (separate flag, off by default).
3. **`SOURCE:` provenance line** at the top of every read verb's text output, **always present, never optional**:
   - `SOURCE: live` when the data was just fetched (`--refresh` path or runtime tier was already expired and auto-refreshed).
   - `SOURCE: cached 47s ago (runtime: 47s, inventory: 3m12s)` when served from cache, with per-tier ages.
   - `SOURCE: cached 47s ago — stale, run with --refresh to re-fetch` when *any* tier is past its TTL but the verb still chose to serve cached (e.g. when refresh fails — see #5).
   - In `--json`, this becomes a structured `source: { mode: "live"|"cached", runtime_age_s: 47, inventory_age_s: 192, stale: false }` field on every read-verb response. Schema goes into `tests/help_json_snapshot.rs` as a contract.
4. **Per-namespace TTL configuration** in `~/.inspect/config.toml`:
   ```toml
   [cache.arte]
   inventory_ttl = "10m"   # default 5m
   runtime_ttl   = "5s"    # default 10s
   ```
   Setting either to `"0"` disables caching for that tier on that namespace (forces every read to be live). Setting to `"never"` disables auto-expiry (only manual `--refresh` invalidates). Default keys (`[cache]` without a namespace) apply to all namespaces; per-namespace overrides win.
5. **Auto-refresh on cache miss / expiry, with graceful degradation:** when a verb runs and the runtime tier is expired, the verb auto-refreshes silently (no operator action needed) and serves live data with `SOURCE: live`. If the auto-refresh **fails** (network error, daemon down), the verb serves the stale cache with `SOURCE: cached … — stale` and a clear chained hint pointing at `inspect connectivity <ns>`. The verb does not error out just because refresh failed — stale data with a loud `stale` flag is more useful than no data, and the operator gets to decide.
6. **Mutation invalidation:** every `--apply` write verb (`restart`, `compose restart`, `bundle run`, the L4 `ssh add-key`) invalidates the runtime tier for the affected namespace at the end of its run. The next read verb on that namespace auto-refreshes — which is exactly the post-restart "did it take?" workflow the 3rd field user needed. Inventory tier is untouched by mutations (containers don't appear/disappear from a restart).
7. **`inspect cache` management verb** (small, lands with F8):
   - `inspect cache show [<ns>]` — print the per-namespace cache state (mtimes, sizes, TTLs, stale-or-fresh) as a table.
   - `inspect cache clear [<ns>]` — wipe the cache for one or all namespaces (forces full refresh on next read).
   - `inspect cache show --json` for agents.

Design points:
- **Atomicity:** cache writes use the existing `write-then-rename` pattern in `src/sys/` so a partial write never poisons the cache.
- **Concurrency:** the runtime tier uses an OS-level advisory lock during refresh to avoid two parallel `inspect` invocations both running `docker inspect` against the same host. The second waits up to 2s for the first's refresh to land, then reads from cache.
- **No background daemon, no watchers, no inotify.** The cache is purely lazy / pull-based; refresh happens on demand at verb invocation. Inspect remains a one-shot CLI.
- **Performance ceiling:** runtime refresh against a 50-container host must complete in ≤ 2s on the same connection profile that already runs `inspect status` today. Test enforces this.
- **Backward compat:** none required — pre-v0.2.0 contract. The cache file format is bumped (new `schema_version: 2`) and v0.1.2 cache files are silently discarded with a one-line debug-log entry on first run.
- The implementation lives in `src/discovery/cache.rs` (new file split out of `src/discovery/engine.rs`) so the tiered logic is unit-testable in isolation. Verbs pull through a single `Cache::get(tier, ns, opts)` function.

**Test:**

Both reflect the cache state correctly (live vs cached vs stale).
- Post-mutation invalidation: mock medium runs `inspect restart arte/onyx-vault --apply`; immediately afterward `inspect why arte/onyx-vault` returns `SOURCE: live` (auto-refreshed) reflecting the new state, **not** the pre-restart cache. This is the 3rd field user's exact failure scenario.

### F9 — `inspect run` silently drops local stdin instead of forwarding it to the remote command
**Source:** v0.1.2 field feedback (3rd field user, BUG-3 follow-up). Real-world session: operator wanted to apply a local Postgres init script to a remote container via `inspect run arte 'docker exec -i atlas-pg sh' < ./01-roles-and-dbs.sh`. `inspect` reported `SUMMARY: run: 1 ok, 0 failed` and exited 0, but **no SQL ran** — the script's stdin never reached the remote `sh`. Workaround required pre-syncing the file to the host with `.deploy.sh` and then running `inspect run arte 'docker exec -i atlas-pg sh < /home/ubuntu/.../01-roles-and-dbs.sh'` (redirect on the remote side, against a remote-side path). Reproducer the user provided: `echo "echo got: $(cat)" | inspect run arte 'sh'` prints `got:` (empty) instead of `got: <stdin>`.
**Severity:** **Medium-high — silent correctness bug on the workhorse verb.** "Exit 0, `1 ok, 0 failed`, command got no input" is the worst possible failure mode: the operator only catches it by externally verifying state afterwards. If the operator is an agent driving a multi-step recipe, this becomes a phantom-success that propagates downstream (e.g. the next step assumes the DB is provisioned and fails for a different reason, masking the root cause). It also blocks the entire "agent has a file in its repo, server has the daemon" idiom — one of the most common one-shot agentic patterns — without an `inspect cp` round-trip. Not a ship-blocker on its own (workaround exists), but lands before all L-items because it undermines confidence in `run`.
**Problem:** `src/commands/run.rs` (and the underlying SSH executor in `src/ssh/`) do not connect the parent process's stdin to the remote command's stdin. The local stdin file descriptor is either dropped at the audit/wrap layer or attached to `/dev/null` when `ssh` is spawned. Native `ssh host cmd` does forward stdin by default; `inspect run` diverges from that contract without documenting it and without warning when local stdin has data waiting (non-tty + readable).
**Fix:** Bring `inspect run` in line with native `ssh host cmd` semantics. Forward stdin by default; opt-out for safety; never silently drop data.

1. **Default behavior change:** when `inspect run`'s own stdin is non-tty (i.e. piped or redirected from a file), forward it byte-for-byte to the remote command's stdin via the existing SSH transport. Close the remote stdin when local stdin reaches EOF, so commands that read until EOF (`sh`, `psql`, `cat`, `tee`, `gpg`) terminate normally. When local stdin **is** a tty, behavior is unchanged from today (no forwarding, no prompt — same as `ssh -T host cmd <terminal>`).
2. **Audit-log the forwarded byte count.** Every `inspect run` invocation that forwards stdin writes a `stdin_bytes: <N>` field into its audit entry. This is the audit contract that makes stdin-forwarding safe to ship: every byte that crossed the boundary is accounted for, and a post-hoc audit can answer "what input did this command consume?" by size. Add `stdin_sha256: "<hex>"` when `--audit-stdin-hash` is set (off by default for perf, opt-in for security-sensitive runs).
3. **Loud failure on the unsafe path.** New flag `--no-stdin` (and config `run.forward_stdin = false` per-namespace) explicitly disables forwarding. When forwarding is disabled **and** local stdin is non-tty with data waiting, `inspect run` exits 2 *before* dispatching the remote command with: `error: stdin has data but forwarding is disabled (--no-stdin or run.forward_stdin = false). → drop the redirect, or pass --stdin to forward, or use 'inspect cp' + remote-side redirect (see 'inspect help run')`. **Never silently discard input.** This is the contract that prevents the field user's exact failure mode regardless of the default.
4. **Size cap with explicit override.** Default cap **10 MiB** of stdin per `inspect run` invocation. Above that, exit 2 with `error: stdin exceeds 10 MiB cap — pass --stdin-max <size> to override, or use 'inspect cp' for large payloads (faster, resumable, audit-tracked separately)`. The cap exists because (a) the audit entry stores the byte count and `inspect cp` is the right tool for bulk transfer with progress + retry, and (b) it bounds memory if the SSH transport ever needs to buffer. Configurable via `--stdin-max <size>` (k/m/g suffixes) and `run.stdin_max = "100m"` per-namespace. Set `run.stdin_max = "0"` to disable the cap entirely (operators who know what they're doing).
5. **Compose-aware:** when `inspect compose restart` (F6) gains any future variant that takes input, it inherits the same forwarding contract. Documented in the compose verb spec.
6. **`inspect help run`** gets a new "stdin handling" section that explicitly covers: default forwarding behavior, the size cap, `--no-stdin`, `--audit-stdin-hash`, the audit fields written, and the "use `inspect cp` for large payloads" guidance. This is the documentation arm of fix #3 from the field user's suggestion list.
7. **Backward compatibility:** none required pre-v0.2.0. The behavior change is from "silently drop" to "forward by default" — this can only improve correctness for any caller that was relying on the drop, because no script intentionally pipes data into `/dev/null` and reports success. CHANGELOG flags this as a behavior change so anyone who *was* relying on the drop sees it.

**Test:**
- `tests/phase_f_v013.rs::run_forwards_local_stdin` — `echo "got: $(cat)" | inspect run <ns> 'sh'` against a mock medium prints `got: hello` (where `hello` is the piped input). This is the field user's exact reproducer; today this test fails before the fix.
- File redirect: `inspect run <ns> 'cat > /tmp/payload' < ./local-file.txt` then `inspect run <ns> 'sha256sum /tmp/payload'` matches `sha256sum ./local-file.txt`. Byte-for-byte fidelity test.
- EOF semantics: `inspect run <ns> 'wc -c' < ./fixture-1234bytes.bin` exits 0 with `1234` on stdout. Confirms remote stdin closes on local EOF.
- TTY default: `inspect run <ns> 'cat'` from an interactive terminal does **not** forward (no hang waiting for input; same behavior as v0.1.2). Detected via `isatty(0)`.
- `--no-stdin` with non-tty input: `echo x | inspect run <ns> --no-stdin 'cat'` exits 2 *before* dispatching SSH, with the chained hint pointing at `inspect cp` + remote-side redirect. Mock medium command counter shows zero remote commands fired (the loud-failure contract).
- Size cap: `dd if=/dev/zero bs=1M count=11 | inspect run <ns> 'wc -c'` exits 2 with the cap error and a hint pointing at `inspect cp`. With `--stdin-max 0`, the same input is forwarded successfully.
- Audit contract: `echo hello | inspect run <ns> 'cat'` writes an audit entry with `stdin_bytes: 6` and (with `--audit-stdin-hash`) `stdin_sha256: "<hex of 'hello\n'>"`. Without `--audit-stdin-hash`, the hash field is absent.

A remote command that exits 0 after consuming a partial-stdin (e.g. SIGPIPE on the writer side) marks the run as failed in the SUMMARY — explicit fix for the silent-success symptom.
- Regression guard: `inspect run <ns> 'echo hi'` with no piped input is unchanged from v0.1.2 (same stdout, same audit entry shape minus the new `stdin_bytes: 0` field).

### F11 — Universal pre-staged `--revert` on every write verb
**Source:** v0.1.2 field feedback (4th field user, ~15-min real ops session). The user's load-bearing observation: *"If I had to pick one feature for AI ops agent safety, it's `inspect revert <audit-id>` — that's the equivalent of `git revert` for production, and it's exactly the missing primitive most 'agent runs SSH' workflows lack."* Today this primitive only exists post-hoc, only on a subset of verbs that emit reversible audit entries, and only by looking up an ID after the fact. The user's stronger formulation is: **every** write verb should be revertible by audit-id, not just the ones that happen to support it today, so that an agent driving a multi-step recipe has a uniform "can apply, can still revert if turns to shit" contract across the entire write surface.
**Severity:** **Load-bearing for agentic safety.** This is the architectural primitive that makes `--apply` safe to grant to a model. Without uniform revert, the operator has to memorize which write verbs are reversible and which aren't, and an agent has no way to ask "can this be undone?" before acting. With uniform revert, the read/write/revert triad becomes a real contract: every byte the tool wrote, it can describe how to un-write. **Non-negotiable before v0.2.0 freezes the audit schema** — adding the inverse-capture fields after the freeze would be a breaking change to every existing audit entry.
**Problem:** Today's `inspect revert <audit-id>` works only for write verbs that opportunistically captured an inverse at apply-time (subset of `restart` paths and `bundle run` steps that happen to declare a `revert:` clause). Other write verbs (`exec`, `ssh add-key`, the L4 password-key migration, F6's future `compose restart`, F8's `cache clear`, anything operators reach for via `--apply`) write an audit entry but no inverse, so `inspect revert <id>` on those entries returns `error: audit entry has no inverse — this verb does not support revert`. That error makes the safety story conditional, which means in practice agents and operators cannot rely on it.
**Fix:** Make inverse-capture a **verb-implementation contract**, not an opt-in feature.

1. **Audit schema additions (lands first; everything else hangs off this).** Every audit entry written by any write verb gains four fields, all required:
   - `revert.kind`: one of `"command_pair"` (run cmd X to undo), `"state_snapshot"` (restore captured state blob), `"composite"` (multi-step inverse, ordered list of the prior two), or `"unsupported"` (verb declares no inverse exists — see #4 below for what triggers this).
   - `revert.payload`: the structured inverse — a remote command string, a base64-encoded snapshot reference, or a list. Schema is closed (`tests/help_json_snapshot.rs` covers it).
   - `revert.captured_at`: timestamp of inverse capture (always **before** the apply step ran — see #2).
   - `revert.preview`: a one-line human-readable description of what the inverse will do (`"restart container atlas-vault to image sha256:abc… (was running before this entry)"`). Used by `inspect revert <id> --dry-run` and by the F11 pre-apply preview.
2. **Capture-before-apply contract.** Every write verb's executor MUST capture the inverse **before** dispatching the mutating action. The order is: (a) compute inverse, (b) write the audit entry with `revert.kind` populated and `applied: false`, (c) dispatch the mutation, (d) update the audit entry to `applied: true` (or `failed: true` with stderr captured). If step (a) fails (e.g. the verb fundamentally cannot describe its inverse on this host), the verb refuses to apply with a loud error pointing at `--no-revert` (see #4); it does not silently apply without an inverse.
3. **`--revert` is now a verb-time flag, not just a separate command.** Every write verb gains:
   - `--revert-preview` — print the captured inverse before applying. Output is the `revert.preview` string plus, under `--debug`, the full `revert.payload`. Composes with `--apply`: `inspect exec arte 'systemctl restart vault' --apply --revert-preview` shows what `inspect revert <new-id>` will do, applies, prints the audit-id at the end.
   - `--revert-on-failure` — if the apply step fails partway through (composite inverse: e.g. step 3 of 5 failed), automatically execute the inverse for the steps that did apply, leaving the system as it was before the verb ran. Off by default (some operators want to inspect a partial state); on-by-default for `bundle run` (which already has rollback semantics — F11 unifies that with the verb-level contract). Audit logs both the original entry (with `failed: true, partial: true`) and the auto-revert entry (linked via `auto_revert_of: <original-id>`).
   - `--dry-run` (alias `--explain`) — render the full effective dispatch plan without executing anything on the remote: the resolved selector, the env-overlay-prefixed remote command line (or, for F14 script mode, the SHA-256 + line count of the script body and the first 5 / last 5 lines), the captured `revert.preview`, and (for F17 multi-step runs) each step's command. Exits 0 with no remote side-effects and no audit entry. **Migration-operator request:** "for destructive ops (`docker volume rm`, `docker compose down -v`, network removes) I would have liked an `inspect run --dry-run` that previewed the script without executing on the remote" — this is the answer for both `run` and every write verb. Mutually exclusive with `--apply`.
4. **`revert.kind: "unsupported"` is reserved for verbs whose effect is intrinsically non-reversible** (e.g. `inspect run` with stdin that mutated remote state via a free-form script — there is no general inverse for an arbitrary `sh` invocation). For those verbs, the operator must explicitly opt in to applying without an inverse: `inspect run --apply --no-revert <selector> '<script>'`. Without `--no-revert`, applying any free-form mutation refuses with: `error: this verb has no general inverse. → pass --no-revert to apply anyway (audit entry will be marked unrevertable), or use 'inspect exec' / 'inspect compose restart' which do capture inverses`. **`inspect run` itself remains read-by-default** (per the existing run/exec split — F11 does not change that); this clause only kicks in if a user has wired `inspect run --apply` via the existing override.
5. **`inspect revert <audit-id>` upgrades:**
   - `inspect revert <id> --dry-run` prints `revert.preview` + `revert.payload` and exits without acting. Default behavior for any `revert` invocation in an interactive tty when the inverse is `composite` (multi-step) — operator must pass `--apply` to actually revert. Non-tty / agent invocations still apply directly.
   - `inspect revert <id>` on an entry with `revert.kind: "unsupported"` exits 2 with the explanation, never silently no-ops.
   - `inspect revert --last [<ns>]` / `--last <n> [<ns>]` revert the most recent N audit entries on a namespace, in reverse chronological order, stopping on the first `unsupported` entry (loud — does not silently skip). This is the "I just made a mess on `arte`, undo the last three things" workflow.
   - `inspect audit show <id>` (existing verb) gains a `REVERT:` block displaying `revert.preview` and `revert.kind`.
6. **Bundle integration.** F6's compose verbs and the existing `bundle run` engine consume the same contract: every step's audit entry has `revert.*` populated, and `bundle run --revert-on-failure` (now default for bundle, was already the bundle behavior) walks the per-step inverses in reverse order. L6 (per-branch rollback in bundle matrix) builds on this — F11 lands the verb-level contract, L6 extends it to matrix branches.
7. **Documentation.** `docs/MANUAL.md` gets a new top-level section "The revert contract" covering: when inverses are captured, what `revert.kind` means, how `--revert-on-failure` interacts with `--apply`, when `--no-revert` is required, and the agent-facing guarantee ("any audit entry with `revert.kind != 'unsupported'` is revertible by id, indefinitely, until the underlying state is independently mutated"). `docs/RUNBOOK.md` gets the verb-author checklist: every new write verb MUST declare its `revert.kind` and a capture function; CI fails (via `tests/no_dead_code.rs`-style scan) if a write verb is registered without one.
8. **Backward compatibility.** Pre-v0.2.0 contract — no compatibility owed. v0.1.2 audit entries lack `revert.*` fields; on read, they are treated as `revert.kind: "unsupported"` and `inspect revert <old-id>` exits 2 with `error: audit entry predates the revert contract (v0.1.2 or earlier) — no inverse captured` and a hint pointing at `inspect audit show <id>` for context. CHANGELOG flags this as a behavior change for the audit log shape.

**Test:**
- `tests/phase_f_v013.rs::every_write_verb_captures_an_inverse` — enumerate the write-verb registry; for each, drive a mock medium through one apply and assert the resulting audit entry has `revert.kind ∈ {"command_pair", "state_snapshot", "composite", "unsupported"}` and a non-empty `revert.preview`. This is the contract-enforcing test that catches a future write verb shipped without a capture function.
- Capture-before-apply: under a mock medium that fails the dispatched mutation, the audit entry exists with `applied: false, failed: true` and `revert.payload` populated (the inverse was captured before the failure). `inspect revert <id>` on that entry no-ops cleanly (nothing to undo, since nothing applied) and emits a one-line note.

- `--revert-on-failure`: a 5-step bundle where step 3 fails leaves the host in the pre-bundle state (steps 1+2 inverses executed in reverse order); audit log contains the original entry + an `auto_revert_of: <id>` linked entry.

- Schema contract: `tests/help_json_snapshot.rs` includes the audit-entry schema with the four new fields; any future change to the schema requires an explicit snapshot update.
- Field-validation gate: end-to-end smoke test against a real multi-container host where an agent runs `inspect compose restart arte/atlas-vault --apply` then immediately `inspect revert --last arte` and the container returns to its prior image+config without operator intervention.

### F12 — Per-namespace remote environment overlay (`PATH`, `LANG`, custom env) applied automatically by `connect`
**Source:** v0.1.2 driving session — LLM-driven `inspect run` invocations on a Codespace-style host repeatedly prepended `PATH="$HOME/.local/bin:$PATH"` to remote commands because the non-login non-interactive SSH shell on that host doesn't pick up `~/.profile` / `~/.bashrc` adjustments that put `~/.local/bin` on PATH. Default Ubuntu setups behave correctly; Codespace-style and other minimally-configured hosts do not. The friction is recurring (every `run` / `exec` invocation needs the prefix) and fully mechanical (the prefix is the same every time), so it should be config, not argv.
**Severity:** Low individually, **high cumulatively for agentic callers.** A model driving the tool either (a) re-derives the prefix every invocation, wasting tokens and inviting drift in how the prefix is spelled, or (b) gets it wrong once and silently runs the wrong binary (e.g. system `python3` instead of the venv-local one). Either failure mode is exactly the kind of recurring papercut that the namespace abstraction is supposed to hide. Lands in v0.1.3 (not v0.1.5) because v0.1.4 is k8s-only and leaving this until v0.1.5 means two release cycles of every model invocation re-prepending the same string.
**Problem:** `inspect connect <ns>` establishes the persistent agent socket and authenticates, but does not negotiate or persist any environment overlay. Every subsequent `inspect run` / `exec` / `compose <verb>` invocation runs in whatever environment the remote SSH non-login shell hands it, which on Codespace-style hosts is missing `~/.local/bin` from PATH. The current workaround — prepending `PATH=…` to every remote command string — is per-invocation, error-prone (one missed invocation runs the wrong binary), and pollutes the audit log with a repeated literal that is really about transport, not intent.
**Fix:** Per-namespace env overlay declared once, applied transparently to every later verb on that namespace.

1. **Config** in `~/.inspect/config.toml` (also writable into `servers.toml` per-namespace):
   ```toml
   [namespaces.arte.env]
   PATH = "$HOME/.local/bin:$PATH"
   LANG = "C.UTF-8"
   # arbitrary string-string pairs allowed
   ```
   The `$HOME` and `$PATH` references on the right-hand side are **resolved on the remote host** at command-dispatch time, not on the client (a literal `$HOME` is sent to the remote shell, which expands it). This is intentional: the operator wants `~/.local/bin` of the remote user, not the local user.
2. **`inspect connect --set-path <value>`** as a one-shot helper for the most common case. Equivalent to writing `PATH = "<value>"` into `[namespaces.<ns>.env]` and saving the config. Also `--set-env KEY=VALUE` (repeatable) for arbitrary entries, and `--unset-env KEY` to remove an entry. Persists immediately; no separate "save config" step.
3. **`inspect connect --detect-path`** runs a one-shot probe on the remote (`bash -lc 'echo $PATH'` vs the current non-login PATH) and, if they differ, **prompts** the operator with a one-line summary: `remote login PATH adds: $HOME/.local/bin, /opt/foo/bin — pin these for arte? [y/N]`. On `y`, writes the diff into `[namespaces.arte.env].PATH`. Off by default; explicitly opt-in (no automatic config writes from `connect`). Designed to make the Codespace-host fix one keystroke from `inspect connect arte --detect-path`.
4. **Application is transparent to the verb layer.** The SSH executor in `src/ssh/` (or wherever `inspect run` / `exec` / `compose` build the remote command line) consults the namespace env overlay and prepends an explicit `env KEY1=VAL1 KEY2=VAL2 …` before the command. **Not** prepended via shell metacharacters that could escape on operator-supplied input — uses the `env` binary's argv form so quoting is safe. The audit entry records the overlay separately (`env_overlay: { PATH: "...", LANG: "..." }`) so an audit reader can see what env was injected without parsing the remote command line.
5. **Operator-side override.** A flag-time override exists for the cases where the namespace overlay is wrong for one invocation: `inspect run --env PATH=... --env-clear arte 'cmd'`. `--env-clear` drops the namespace overlay entirely for this invocation; `--env KEY=VALUE` (repeatable) merges on top of the namespace overlay (or, with `--env-clear`, replaces it). The audit entry records both the namespace overlay and the per-invocation override so the effective env is reconstructable.
6. **Visibility.** `inspect connect --show` (and `inspect status <ns>`'s extended output) print the active env overlay for the namespace. `inspect run --debug <ns> 'cmd'` prints the remote command line as actually dispatched (including the `env` prefix) so the operator can confirm what ran. **No silent env injection** — every overlay is either printable on demand or visible in the audit entry.
7. **Out of scope for F12 (deferred to v0.1.5):** secret-backed env values (e.g. `PATH = !ref:keychain:arte-path`), per-verb overlays (different env for `exec` vs `run`), and computed values (e.g. `now()` timestamps). v0.1.3 ships static string-string pairs only; this is enough to fix the Codespace PATH friction and 90% of the same-shape future requests, without committing to a templating engine that L3 (parameterized aliases) is the better venue for.
8. **Documentation.** `docs/MANUAL.md` gets a "Per-namespace environment overlay" section under the connect docs, including the Codespace-style worked example (`inspect connect arte --detect-path` → operator confirms → from then on `inspect run arte 'which inspect-aux'` finds the local-bin binary without a prefix). `docs/RUNBOOK.md` gets the dispatch contract: env overlay is applied by the SSH executor, not by individual verbs, so any new verb inherits the behavior automatically.

**Test:**
- `tests/phase_f_v013.rs::env_overlay_applied_to_run_and_exec` — with `[namespaces.<mock>.env].PATH = "/extra/bin:$PATH"`, `inspect run <mock> 'echo $PATH'` returns a string starting with `/extra/bin:`. Same for `inspect exec <mock> --apply 'echo $PATH'`.
- Audit-entry shape: the recorded entry includes `env_overlay: { PATH: "/extra/bin:$PATH" }` as a structured field, **and** the rendered remote command (recorded for replay) starts with `env PATH=/extra/bin:$PATH …`.
- `--env-clear` drops the overlay for that invocation only: `inspect run --env-clear <mock> 'echo $PATH'` returns the remote shell's default PATH, then a subsequent `inspect run <mock> 'echo $PATH'` (no flag) returns the overlaid PATH again.

Test asserts the remote `printenv MALICIOUS` echoes the literal `;` back.

### F13 — Stale-session auto-reauth: re-prompt for key passphrase in-place on the failing verb
**Source:** v0.1.2 field feedback (recurring across multiple sessions, **explicitly re-flagged as the #1 pain point by the migration operator** after a multi-hour destructive migration: "`inspect run arte -- 'echo hello'` returned `arte: exit 255 / SUMMARY: run: 0 ok, 1 failed` with **no hint** that the cause was a dead socket. I had to ask the user to disconnect/reconnect because the error code is the same as 'command actually failed.'"). When the persistent agent socket / `ControlPersist` master connection for a namespace expires (TTL elapsed, remote-side `sshd` killed it, network blip closed the channel), the next `inspect run` / `exec` / `logs` / any-verb-that-dispatches errors with an SSH-level "broken pipe" or "connection closed" and exits non-zero. The operator's only recovery path today is the manual two-step `inspect disconnect <ns> && inspect connect <ns>` cycle (which re-prompts for the key passphrase), then re-running the original verb. For agentic callers driving a long-lived session this is worse: the model sees a non-zero exit, has no signal that the failure was transport-level rather than command-level, and either bails the recipe or retries the same verb against the same dead socket and fails again.
**Severity:** Medium. Not a correctness bug (the verb really did fail, exit code is honest), but it converts a routine session-expiry — which happens silently on every host with a non-trivial `ClientAliveInterval` / agent TTL — into a manual interruption that disproportionately hurts long-running ops sessions and any agent-driven multi-step recipe. Two-line fix at the dispatch layer (detect + retry once with re-auth) eliminates the friction without any new surface area for operators to learn.
**Problem:** The SSH executor in `src/ssh/` treats every dispatch failure as terminal. There is no classification step that distinguishes "transport-level: master socket / agent expired, re-auth and try again" from "command-level: the remote command actually failed". As a result the stale-session case bubbles up as a verb failure even though the verb itself never executed remotely. The `inspect connect` flow knows how to prompt for the passphrase and re-establish the master socket; that flow is just not wired into the verb-time failure path.
**Fix:** Detect transport-level staleness on dispatch failure and transparently re-auth on the same invocation. Single retry, never silent.

1. **Classify dispatch errors at the SSH executor boundary** (`src/ssh/`): split the existing single `error` return into:
   - `Transport::Stale` — master socket gone / `ControlPersist` expired / "broken pipe" / "connection closed by remote host" before any command bytes were read. Detected by stderr-pattern match against the known set of openssh strings, by errno on the control-socket file, and by the `ssh -O check` probe returning non-zero.
   - `Transport::Unreachable` — host genuinely unreachable (DNS fail, connection refused, network down). No re-auth helps; bubble up with the existing chained hint pointing at `inspect connectivity <ns>`.
   - `Transport::AuthFailed` — re-auth attempt itself failed (wrong passphrase, key revoked). Bubble up with a chained hint pointing at `inspect connect <ns>` for the operator to debug interactively. **Never** silently retry a failed re-auth (no infinite-loop risk).
   - `Command::Failed` — remote command exited non-zero. Verb-level failure, no transport involvement. Today's behavior unchanged.
2. **On `Transport::Stale`, the executor:**
   - Emits a single one-line stderr notice on the failing verb: `note: persistent session for arte expired — re-authenticating…` (always visible; not gated behind `--debug` because the operator needs to know a passphrase prompt is about to appear and why).
   - Invokes the same re-auth path that `inspect connect <ns>` uses, including the F7.6 line-ordering fix (no spurious `Permission denied` lines before the passphrase prompt).
   - On successful re-auth, **re-dispatches the original verb exactly once** with the new master socket. Subsequent failure (transport or command) is reported as the verb's failure, not retried again. One retry per verb invocation is the contract — no exponential backoff, no pool, no daemon.
   - On failed re-auth (wrong passphrase, key not loadable), surfaces `Transport::AuthFailed` with a chained hint and the original verb is **not** retried.
3. **Audit-log the re-auth event.** Every transparent re-auth writes a separate audit entry (`verb=connect.reauth, trigger=transport_stale, namespace=<ns>, original_verb=<verb>, original_audit_id=<id>`). The original verb's audit entry gains a `reauth_id: <connect.reauth-id>` field linking to the re-auth. This keeps the audit log honest: every passphrase prompt the tool issued is recorded, and post-hoc readers can see that `inspect run arte 'foo'` actually involved a re-auth midway. The retry of the original verb writes its own audit entry as normal, with `retry_of: <original-id>` so a reader can see the relationship.
4. **Non-tty / agent invocations.** If local stdin is not a tty (the verb is being driven by an agent, a script, or a CI step), there is no operator to type a passphrase. In that case the re-auth path:
   - Tries the ssh-agent first (no prompt — agent already has the key cached if the session was set up with it). If that succeeds, the retry proceeds silently except for the one-line stderr notice.
   - If the agent has no usable key, refuses with `error: persistent session for arte expired and re-auth requires a passphrase but stdin is not a tty. → run 'inspect connect arte' interactively, then re-invoke this command, or load the key into ssh-agent so future re-auths are non-interactive`. Exit 2. **Never** block waiting for input on a non-tty (which would hang an agent indefinitely).
5. **Per-namespace opt-out.** Default behavior is auto-reauth on stale; for operators who explicitly want the old behavior (e.g. they want to know about every session expiry as a hard failure), `[namespaces.<ns>].auto_reauth = false` in `~/.inspect/config.toml` disables the retry. Per-invocation override `--no-reauth` does the same for one verb. Defaulting to on because the friction it removes is much larger than the loss-of-signal it costs (and the audit log preserves the signal anyway).
6. **Distinct exit class on transport failure, even when auto-reauth is disabled or fails.** This is the migration-operator's load-bearing ask: today, transport failure and command failure both exit non-zero with the same SUMMARY shape, so a wrapper agent cannot distinguish "the SSH connection died, retry me" from "the remote command really failed, do not retry me blindly". Fix: every `Transport::Stale` / `Transport::Unreachable` / `Transport::AuthFailed` failure exits with a **dedicated exit code** (proposed: `12` = stale, `13` = unreachable, `14` = auth-failed; out of the way of conventional `1`/`2`/`126`/`127` and not collidng with remote command exit codes which pass through unchanged for `Command::Failed`). The SUMMARY trailer reads `SUMMARY: run: 0 ok, 1 failed (ssh_error: stale connection — run 'inspect disconnect <ns> && inspect connect <ns>' or pass --reauth)` instead of the bare `0 ok, 1 failed`. The structured `--json` output gains an explicit `failure_class: "transport_stale" | "transport_unreachable" | "transport_auth_failed" | "command_failed" | "ok"` field on every verb response (schema enforced by `tests/help_json_snapshot.rs`). **This contract holds whether or not auto-reauth is enabled** — if auto-reauth is on and succeeds, the retry's exit code is what the verb returns; if auto-reauth is disabled (`--no-reauth`) or fails, the dedicated exit code + chained hint surfaces so an agent or operator can react correctly. Without this, F13's auto-reauth covers the happy path but leaves the disabled / failed-reauth path indistinguishable from a command failure — which is the exact symptom the migration operator hit.
7. **Composes with F12 (env overlay).** The retried verb inherits the same env overlay as the original — the executor's overlay-application step happens after the master socket is established, so re-auth and overlay are independent.
8. **Composes with F11 (revert contract).** A write verb that hit `Transport::Stale` **before** the remote command actually started has `applied: false` on its audit entry (no inverse needed because nothing applied). The retry, if it succeeds, writes a fresh audit entry with the captured inverse and `applied: true`. A write verb that hit `Transport::Stale` **after** dispatching but before reading the exit status is rare but possible — handled conservatively: the retry is **not** automatic for write verbs in that ambiguous state; instead the executor surfaces `error: connection lost while write verb was in flight — state on remote is unknown. → run 'inspect status <ns>' to verify, then re-invoke or 'inspect revert <id>' as appropriate`. Read verbs auto-retry unconditionally (idempotent); write verbs auto-retry only when the dispatch demonstrably failed before remote execution started.
9. **Documentation.** `docs/MANUAL.md` "Connections" section gets a paragraph on auto-reauth behavior, the one-line notice operators will see, the `--no-reauth` opt-out, and the non-tty failure mode. `docs/RUNBOOK.md` gets the executor-classification table (the four `Transport::*` / `Command::*` categories) so the next SSH-executor maintainer knows where to extend the pattern matchers.

**Test:**

Stderr contains the one-line `note: … re-authenticating…` exactly once. Audit log contains both the `connect.reauth` entry and the retry entry linked by `retry_of` / `reauth_id`.
- Read-verb retry is unconditional: same test against `inspect status <ns>` succeeds.
- Write-verb pre-dispatch failure auto-retries: `inspect exec <ns> 'cmd' --apply` where the staleness is detected before any remote bytes are sent re-auths and retries cleanly.
- Write-verb mid-flight failure does **not** auto-retry: inject a connection drop after the remote command started; executor returns `error: connection lost while write verb was in flight …` with the chained hints, exit 2, no automatic retry. Audit entry has `applied: unknown` and a clear marker.
- Non-tty path with no agent key: same staleness scenario but `isatty(0)` returns false and the mock ssh-agent has no usable key; verb exits 2 with the chained hint, **no** prompt-for-passphrase blocking.

- `Transport::AuthFailed` is **not** retried: after one failed re-auth attempt with a wrong passphrase, the executor exits 2 with the chained hint, never loops.
- `Transport::Unreachable` is **not** retried with re-auth: DNS-fail and connection-refused cases bubble the original error with the `inspect connectivity <ns>` hint.

Same shape (codes `13` / `14`, classes `"transport_unreachable"` / `"transport_auth_failed"`) for the other two transport categories. Genuine command failure (`Command::Failed`) preserves the remote command's exit code unchanged, SUMMARY does **not** contain `ssh_error:`, and `failure_class` is `"command_failed"`. This is the migration-operator-reported reproducer test: `inspect run arte -- 'echo hello'` against a deliberately-killed master socket exits 12 (not 255 with no hint) and emits the chained-hint trailer.
- Audit-log linkage: an end-to-end run that triggers re-auth on `inspect run arte 'cmd'` produces three linked entries (`run` original with `reauth_id`, `connect.reauth`, `run` retry with `retry_of`) and `inspect audit show <run-original-id>` displays all three in the new "REAUTH:" block.

### F14 — `inspect run --file <script>` / `--stdin-script` / heredoc-on-stdin script mode
**Source:** Migration-operator field feedback after a multi-hour destructive migration on a 60-image host: *"The biggest individual time-sink was nested quoting. Every layer (your shell → ssh → bash → docker exec → psql/python -c) needs its own escape pass… I'd pay almost any feature in tradeoffs to never quote-escape across layers again."* The `inspect run arte -- '<heredoc>'` flow is already the killer feature of the tool (the operator's words); the only thing standing between it and "this is the way I touch the server" is that a non-trivial bash payload with embedded `psql -c "..."`, `python -c "..."`, and `cypher-shell ... <<CYPHER` requires the operator to count and balance quote characters across four shell layers.
**Severity:** Medium-high for any operator running multi-step heredocs; **load-bearing for migrations**, which is precisely the workflow `inspect run` was designed to enable. F14 turns the existing argv-string interface into a file/stdin interface so the script body never crosses a shell-parsing boundary on the local side.
**Problem:** Today `inspect run <ns> -- '<script>'` takes the script as a single argv token. That token has already been parsed by the operator's local shell once (which consumed one layer of quoting), then it crosses the SSH boundary as a single argument to the remote `bash -c <token>` (which parses it again). The operator's only recourse is to manually escape every embedded quote, every `$`, and every `\` in a way that survives both passes \u2014 a constraint that does not scale to the multi-tool migration heredocs the tool's structured output (`arte | …`, `SUMMARY: …`) is otherwise *built* to support. F9 (stdin forwarding) is necessary infrastructure but not sufficient: F9 forwards local stdin to the *remote command's* stdin, while F14 wants the operator's local script body to *become* the remote command body.
**Fix:** Three new entry points to `inspect run` (and `inspect exec` once it gains `--apply`), all of which dispatch via `bash -s` on the remote so the script body is read from remote stdin rather than parsed as argv \u2014 zero shell parsing of the script on the local side beyond reading the file.

1. **`inspect run <ns> --file <path>`** \u2014 reads `<path>` from the local filesystem and ships its contents as the remote command body via `bash -s` (or `sh -s` if the script's shebang declares a different interpreter; F14 honors a `#!/usr/bin/env <interp>` first line and dispatches accordingly). The script body is **never** parsed by any local shell beyond the one that already invoked `inspect`. Composes with F9: arguments after `--` are forwarded as `bash -s -- arg1 arg2 …` so the script can use `$1` / `$2` normally.
2. **`inspect run <ns> --stdin-script`** \u2014 reads the script body from `inspect`'s own stdin (`isatty(0)` must be false, otherwise exits 2 with a hint pointing at `--file`). This is the heredoc form: `inspect run arte --stdin-script <<'BASH' … BASH`. Mutually exclusive with F9's stdin-forwarding-to-remote-command behavior \u2014 when `--stdin-script` is set, local stdin becomes the *script source* and the remote command's stdin is `/dev/null`. Operators who need both (script body + remote-command stdin) use `--file` for the script and pipe stdin normally (F9 forwards it to the script's `stdin`).
3. **Bare `--`-followed-heredoc still works.** `inspect run <ns> -- 'echo one-liner'` is unchanged. The new flags are opt-in for the multi-line case.
4. **Audit-log the script body, not just a hash.** Migration audit trails need the actual script for post-mortem reconstruction; F18 (session transcript) covers verbose capture, but the audit entry itself records `script_sha256: "<hex>"`, `script_bytes: <N>`, and (under `--audit-script-body`, off by default for size) the full body inline. Without `--audit-script-body`, the script is stored once per unique sha256 in `~/.inspect/scripts/<sha256>.sh` (deduped, gc'd by F18's rotation policy) and the audit entry references it by hash. This way every audited write-via-script is reconstructable by hash even if the operator has long since deleted the local file.
5. **Composes with F11 (revert contract).** A script-mode invocation has `revert.kind: "unsupported"` by default (free-form scripts are not generally invertible) and therefore requires `--no-revert` if dispatched via `inspect exec --apply --file …`. The exception: if the script declares a `# inspect-revert: <inverse-script-path>` directive in its first 10 lines, F11 captures *that* file's contents as the inverse and `revert.kind` becomes `"command_pair"`. Documented in MANUAL.md so migration scripts can opt in to revertability. **`inspect run --file` itself remains read-by-default**, same as bare `inspect run` (per the F11 run/exec split).
6. **Composes with F12 (env overlay) and F13 (auto-reauth).** Script-mode invocations dispatch through the same SSH executor as bare `inspect run`, so the env overlay is applied (the script sees the configured `PATH`) and stale-session re-auth fires identically.
7. **Size cap.** Script-mode payloads share F9's `--stdin-max` cap (default 10 MiB), since they travel through the same byte path. Above the cap, exits 2 with a hint pointing at F15's `inspect put` for bulk transfer + remote-side execution.

**Test:**
- `tests/phase_f_v013.rs::run_file_dispatches_via_bash_s` \u2014 a `--file` invocation against a 30-line script with embedded `psql -c "SELECT 1"` and `python -c 'print("hi")'` runs end-to-end with **zero local quoting** and the embedded quote characters survive byte-for-byte to the remote interpreters. The migration-operator's exact pain reproducer.
- `--stdin-script` heredoc form: `cat fixture.sh | inspect run <ns> --stdin-script` produces identical output to `inspect run <ns> --file fixture.sh`.
- Shebang dispatch: a script starting with `#!/usr/bin/env python3` is dispatched via `python3 -` instead of `bash -s`. Without a shebang, `bash -s` is the default.
- Audit shape: every `--file` / `--stdin-script` invocation writes `script_sha256` and `script_bytes` into the audit entry; with `--audit-script-body`, the full body is inline. Without `--audit-script-body`, the body is dedup-stored in `~/.inspect/scripts/<sha>.sh` and `inspect audit show <id>` retrieves it from there.
- Revert directive: a script with `# inspect-revert: ./inverse.sh` in its first 10 lines, dispatched via `inspect exec --file --apply`, produces an audit entry with `revert.kind: "command_pair"` and `revert.payload` referencing the inverse script's sha256. Without the directive, `--apply` requires `--no-revert`.
- Mutual exclusion: `--stdin-script` with a tty stdin exits 2 with the `--file`-pointing hint. `--file` and `--stdin-script` together exit 2.
- Args after `--`: `inspect run <ns> --file s.sh -- alpha beta` runs `bash -s -- alpha beta` on the remote and the script's `$1` / `$2` are `alpha` / `beta`.
- Size cap interaction: an 11 MiB script exits 2 with the cap error and the F15 `inspect put` hint.
- Field-validation gate: a real migration heredoc from the 60-image session (preserved as `tests/golden/migration-heredoc-001.sh`) runs end-to-end via `--file` against a mock medium and produces the expected per-step `arte | …` prefixed output without any escape pass on the local side.

### F15 — `inspect put` / `inspect get` native file transfer over the persistent socket
**Source:** Migration-operator field feedback: *"I had to context-switch between `inspect run arte -- 'cat > /tmp/x.sh'` (works for tiny payloads, breaks on quoting) vs `scp /tmp/x ...:arte:/home/ubuntu/...` (works but bypasses the inspect connection). For a tool whose job is 'be the way I touch this server,' not having `inspect put arte LOCAL REMOTE` / `inspect get arte REMOTE LOCAL` is a notable hole. Especially during compose-file rewrites."* The hole is also already implicit in the v0.1.3 design: F9's error hints already direct operators to "use `inspect cp`" for large payloads, and F14's size-cap hint does the same \u2014 F15 is what those hints actually point at. Without F15, those hints point at vapor.
**Severity:** Medium. Not a correctness bug, but the workaround (raw `scp`) bypasses every contract the rest of the tool establishes: no audit log, no env overlay, no auto-reauth, no revert capture, no namespace selector. Any compose-file rewrite, log retrieval, certificate ship-up, or backup pull during an ops session today either round-trips through `inspect run -- 'cat > …'` (quoting-fragile, capped at small sizes) or escapes to `scp` (untracked).
**Problem:** No file-transfer verb exists. Operators reach for `scp` or `rsync` directly, which (a) requires them to know the underlying SSH host string (defeating the namespace abstraction), (b) does not reuse the persistent `ControlPersist` master (extra auth round-trip), and (c) leaves the audit log silent on what files entered or left the host.
**Fix:** Two new verbs that ride on the existing persistent socket and inherit every namespace-level guarantee.

1. **`inspect put <ns> <local-path> <remote-path>`** \u2014 uploads a local file (or directory under `--recursive`) to the remote path. Implemented via `scp` invoked with `-o ControlPath=~/.inspect/sockets/<ns>.sock` so the existing master is reused (no new auth, no passphrase prompt). Supports `--mode <octal>` (chmod after upload), `--owner <user[:group]>` (chown after upload, requires remote root via the namespace's existing privilege model), and `--mkdir-p` (create parent directories on the remote if they do not exist).
2. **`inspect get <ns> <remote-path> <local-path>`** \u2014 downloads. Same flag set minus the chmod/chown options. **`--since <duration>` and `--max-bytes <size>` deferred to v0.1.5** (log-retrieval ergonomics; `inspect logs --since` already covers that domain in v0.1.3, and the migration-operator's pain point is "I cannot transfer files at all", not "I cannot filter downloads"). Bare `get` ships in v0.1.3.
3. **Selector forms.** Both verbs accept the existing selector syntax for the remote side: `inspect put arte/atlas-vault:/etc/vault/config.hcl ./vault-config.hcl` (uploads into a container's filesystem via `docker cp` rather than the host filesystem; F15 detects the container selector and dispatches accordingly), and the host-path form `arte/_:/path` from F7.2. Host vs container filesystem is unambiguous from the selector form, never from a flag.
4. **Audit-log every transfer.** Each `put` / `get` writes an audit entry with `direction: "up"|"down"`, `local_path`, `remote_path`, `bytes`, `sha256` (computed during transfer; the `xfer` machinery streams through a hashing tee), `duration_ms`. **F15 transfers are revertible** under F11: an `inspect put` captures the prior remote-file contents (or absence) as `revert.kind: "state_snapshot"` so `inspect revert <id>` restores. An `inspect get` captures nothing on the remote (it is a read) and has `revert.kind: "unsupported"` (revertable in the trivial sense \u2014 just delete the local file \u2014 but F11 does not capture local-side state). Documented.
5. **Progress + resume.** For files larger than 10 MiB, `inspect put` / `get` prints a one-line progress bar to stderr (suppressed under F7.4 `--quiet`). Resumable transfers via `--resume` are **out of scope for v0.1.3** and explicitly deferred to v0.1.5; the mention here is to head off scope creep during implementation.
6. **Composes with F12 (env overlay) and F13 (auto-reauth).** Same SSH executor path; same guarantees.
7. **Documentation.** New `docs/MANUAL.md` section "File transfer" with the four worked examples (host upload, host download, container upload via `docker cp`, log retrieval via `--since`). The `inspect cp` token used in F9 / F14 hints is documented as an alias for `inspect put` (operators with rsync muscle memory; the verb's canonical name is `put`).

**Test:**
- `tests/phase_f_v013.rs::put_get_roundtrip_byte_for_byte` \u2014 upload a 1 MiB binary fixture; `inspect get` it back to a different local path; `sha256sum` matches the original. Audit entries on both sides recorded with matching sha256.
- Container-filesystem upload: `inspect put arte/atlas-vault:/etc/vault/config.hcl ./fixture.hcl` against a mock medium dispatches via `docker cp` (not host-filesystem `scp`); the file appears inside the container at the correct path.
- Host vs container disambiguation: `arte:/path` (no slash before `:`) is rejected with the F7.2 hint; `arte/_:/path` is host-fs; `arte/<svc>:<path>` is container-fs.
- Revert: `inspect put arte/_:/etc/conf.d/foo ./new.conf` against a host where `/etc/conf.d/foo` exists with prior content writes a `revert.kind: "state_snapshot"` audit entry; `inspect revert <id>` restores the prior content. Against a host where the file did not exist, revert deletes the file.
- Auto-reauth interaction: a stale-socket `inspect put` triggers F13's auto-reauth and the transfer succeeds on retry.
- ControlPath reuse: with the persistent socket warm, `inspect put` does **not** prompt for a passphrase (verified by mocking the prompt path and asserting it is unreached).
- `--mkdir-p`: target dir does not exist; without `--mkdir-p`, exits with `error: remote parent dir does not exist`; with `--mkdir-p`, succeeds and records the dir creation in the audit entry as part of the state snapshot.
- Field-validation gate: end-to-end smoke test of a compose-file rewrite during a real ops session \u2014 `inspect get arte/_:/etc/compose/atlas.yml ./atlas.yml`, edit locally, `inspect put` it back, audit log shows both entries with matching sha256s and the prior version is recoverable via `inspect revert`.

### F16 — `inspect run --stream` / `--follow` line-streaming for long-running remote commands
**Source:** Migration-operator field feedback: *"`docker compose up -d --force-recreate aware aware-embedder 2>&1 | tail -30` is fine, but I'd kill for `inspect run --stream arte -- 'docker logs -f aware'` that line-streams back to the client until I Ctrl-C. I worked around with `inspect logs arte/<service>` (which exists) but it's a separate command shape and doesn't compose with arbitrary scripts."*
**Severity:** Small/medium. The dedicated `inspect logs` verb already covers single-container log tailing; F16 is for the long tail of *other* long-running commands (`tail -f /var/log/syslog`, `journalctl -fu vault`, `python -m monitor`, anything that produces output indefinitely until SIGINT). Today every such command either buffers until exit (silent until the operator gives up and Ctrl-C's the local `inspect`, which often kills the process locally without notifying the remote) or works only by accident if the remote happens to flush eagerly.
**Problem:** The SSH executor consumes remote stdout/stderr in a single read-to-EOF pass, then renders the captured buffer with `arte | …` prefixing. For commands that never EOF until SIGINT, this means zero output appears locally until the operator interrupts \u2014 and the interrupt is currently not forwarded to the remote process, so the remote keeps running detached. F9 (stdin forwarding) and F13 (transport classification) provide the substrate; F16 wires it together for the streaming case.
**Fix:** A `--stream` (alias `--follow`) flag on `inspect run` (and `inspect exec` once it gains `--apply`) that switches to incremental dispatch.

1. **Line-buffered local rendering.** With `--stream`, every newline-terminated chunk from remote stdout/stderr flushes to local stdout immediately with the existing `<ns> | …` prefix. No buffering beyond what the local terminal does.
2. **SIGINT forwarding.** Local `Ctrl-C` (`SIGINT`) is forwarded to the remote process via the SSH session's signal-forwarding channel (`ssh -o RemoteForward` is not used; instead the executor uses the OpenSSH protocol's `signal` request). On receipt of the second `Ctrl-C` within 1 second, the executor escalates to closing the channel locally (the remote process gets SIGHUP from sshd as the channel closes). The operator never types `Ctrl-C` and is left wondering whether the remote process is still running.
3. **Trailer suppression.** With `--stream`, the `SUMMARY:` / `NEXT:` trailer prints **once** at the end after the remote process exits (or the operator interrupts), so `--stream` composes with `--quiet` (F7.4) the same way bare `run` does. The exit code is the remote process's exit code, with the F13 transport-failure exit codes overriding when applicable.
4. **Mutually exclusive with F14's `--stdin-script` for now.** Streaming a script-source over local stdin while also streaming output back is a half-duplex protocol headache that does not need to ship in v0.1.3; document the limitation. `--stream --file <script>` is fine (script body is not streamed; it is delivered in one shot, then the running script's output streams back).
5. **Composes with F18 (session transcript).** Streamed output is appended to the transcript line-by-line as it arrives, not at the end \u2014 so a transcript-driven post-mortem of an interrupted long-running command shows the partial output that was captured before SIGINT.
6. **`inspect logs` interop.** F16 does **not** replace `inspect logs --follow` \u2014 the dedicated logs verb keeps its existing semantics (selector-aware, source-tier-aware, structured). F16 is for the case when the operator wants streaming for a non-logs command (or a multi-step script that includes a streaming step). Documented in F10.6 (logs discoverability).

### F17 — `inspect run --steps <file.json>` multi-step runner with per-step exit codes
**Source:** Migration-operator field feedback: *"When I run a 5-step heredoc, all 5 steps are one 'run' with one exit code. If step 3 fails I see it in the output but the SUMMARY still says `1 ok` because the **outer ssh** succeeded. That made me build defensive `set +e; … || echo MARKER` patterns. A `--steps` mode that took an array of commands and returned per-step exit codes would be amazing for migration scripts."*
**Severity:** Medium. The defensive `set +e; … || echo MARKER` pattern is already widespread among multi-step heredoc users; F17 promotes it from a workaround to a first-class verb mode with structured output that an LLM-driven wrapper can reason about step-by-step. Without F17, agentic callers cannot reliably build "run these N steps, stop on first failure, give me the per-step result table" workflows on top of `inspect run`.
**Problem:** `inspect run` reports a single exit code (the outer `bash -c`'s) and a single `SUMMARY: run: 1 ok, 0 failed` trailer regardless of how many discrete operations the script body performed. A 5-step migration script where step 3 fails reports `1 ok` because `bash -c` exited 0 (`set -e` was off); even with `set -e`, the SUMMARY does not say *which* step failed without the operator parsing the prefixed output by hand. For human operators this is recoverable; for an LLM agent driving multi-host migrations it is not.
**Fix:** A new `--steps <file.json>` mode that takes a structured step list and returns a structured per-step result table.

1. **Input format.** A JSON file (or `-` for stdin) of the form:
   ```json
   {
     "steps": [
       {"name": "snapshot-volumes",   "cmd": "docker run --rm -v vault_data:/d alpine tar czf - /d > /tmp/vault.tgz", "on_failure": "stop"},
       {"name": "stop-vault",          "cmd": "docker compose stop vault",                                              "on_failure": "stop"},
       {"name": "migrate-vault-data",  "cmd": "/usr/local/bin/migrate-vault.sh",                                        "on_failure": "stop", "timeout_s": 600},
       {"name": "start-vault",         "cmd": "docker compose start vault",                                             "on_failure": "stop"},
       {"name": "verify-vault",        "cmd": "docker exec vault vault status",                                         "on_failure": "continue"}
     ]
   }
   ```
   Each step has `name` (required, used in output and audit), `cmd` (required, dispatched via `bash -c` on the remote with the F12 env overlay applied), `on_failure` (`"stop"` default, `"continue"` to record the failure and proceed), and optional `timeout_s` (per-step wall-clock limit, default unlimited). YAML input is accepted via `--steps-yaml` for operators with that preference.
2. **Per-step structured output.** Each step's stdout/stderr is captured separately and emitted under a `STEP <name> ▶` / `STEP <name> ◀ exit=<code> duration=<ms>` block, with the existing `<ns> | …` line-prefixing inside each block. After all steps run (or the first `stop`-on-failure aborts), a structured summary table prints:
   ```
   STEPS: 5 total, 4 ok, 1 failed
     ✓ snapshot-volumes      exit=0   duration=12.3s
     ✓ stop-vault            exit=0   duration=1.1s
     ✗ migrate-vault-data    exit=1   duration=8.7s   (stopped pipeline)
       skipped:               start-vault, verify-vault
   ```
3. **Structured `--json` output.** With `--json`, the entire step list is emitted as a structured object: `{steps: [{name, cmd, exit, duration_ms, stdout, stderr, status: "ok"|"failed"|"skipped"}], summary: {total, ok, failed, skipped, stopped_at: <name>|null}}`. Schema enforced by `tests/help_json_snapshot.rs`. **This is the contract that LLM-driven wrappers can reason about** \u2014 no prose parsing, no defensive markers.
4. **Audit-log per step.** Every step writes its own audit entry, linked by a `steps_run_id: <uuid>` field. The parent `run --steps` invocation also gets an entry that records `steps_run_id` + the full step list manifest (sha256 of the input file). Post-hoc, `inspect audit show <steps_run_id>` displays the full table; `inspect audit show <step-id>` zooms to one step.
5. **Composes with F11 (revert contract) and F14 (script mode).** Each step's audit entry has its own `revert.*` capture (composite verbs like `compose stop` get `command_pair`; arbitrary `bash -c` strings get `unsupported` unless the step declares `"revert_cmd": "<inverse>"` in the step JSON). The parent `--steps` entry has `revert.kind: "composite"` with `revert.payload` = the ordered list of per-step inverses, so `inspect revert <steps_run_id>` walks them in reverse. **`inspect run --steps --apply --revert-on-failure`** unwinds the steps that ran when one fails \u2014 exactly the migration-operator's missing primitive. Also: a step's `cmd` may itself be a `--file` reference (`{"cmd_file": "./step3.sh"}`) so F14 and F17 compose for multi-step migrations whose individual steps are non-trivial scripts.
6. **Composes with F12 (env overlay) and F13 (auto-reauth).** Each step inherits the namespace env overlay; a stale-socket failure mid-pipeline triggers F13's auto-reauth and the failing step retries (subject to `on_failure` semantics).
7. **Composes with F16 (streaming).** `--steps --stream` flushes per-step output line-by-line as it arrives instead of buffering until each step completes; the per-step block markers still print at start and end. Useful for migrations where the operator wants to watch a long step (`migrate-vault-data`) live.

**Test:**

- `on_failure: "continue"` runs all steps even when one fails; SUMMARY reports the count correctly.
- `timeout_s`: a step that sleeps longer than its declared timeout is killed with SIGTERM (then SIGKILL after 5s grace), the audit entry records `exit: -1, status: "timeout"`, and the pipeline behaves per `on_failure`.
- `inspect run --steps … --json` against a 5-step manifest produces a single JSON object whose `steps` array length equals 5, whose `summary` object's counts are correct, and whose `stopped_at` is `null` when all steps run or the failed step's `name` when one stops the pipeline.

### F18 — Session transcript to `~/.inspect/history/<ns>-YYYY-MM-DD.log`
**Source:** Migration-operator field feedback: *"After a 4-hour migration I had to scroll my chat history to find which exact `inspect run` command did what. An `~/.inspect/history/arte-2026-04-28.log` (rotating, all in/out) would let me audit and post-mortem migrations cleanly. **High value for compliance / audit trails of destructive ops.**"*
**Severity:** Medium. The structured audit log already exists (`~/.inspect/audit/`) and answers "what verbs ran with what arguments + what changed", but it is consciously *structured* and not optimized for human linear reading during a post-mortem. F18 ships a complementary human-readable transcript that captures the full input + output of every verb invocation as an operator would have seen it on the terminal \u2014 the equivalent of `script(1)` or `tmux capture-pane` but scoped per-namespace and rotated daily.
**Problem:** Today, after a long ops session, the only sources of truth are (a) the operator's terminal scrollback (lost on terminal close, lost on Codespace rebuild, never queryable), and (b) the structured audit log (queryable but doesn't show interleaved stdout/stderr the way the operator saw it). F18 fills the gap: a per-namespace, per-day file that contains every verb invocation with its full output, in the order they happened, suitable for `grep` / `less` / sharing with a teammate during a post-mortem.
**Fix:** Per-namespace rotating transcript files, on by default, off-able per-namespace.

1. **Path:** `~/.inspect/history/<ns>-<YYYY-MM-DD>.log`. New file each calendar day per namespace; appends within the same day. Permissions `0600`. UTC-anchored (operator-time-zone-independent).
2. **Format:** simple structured plaintext optimized for `grep` and `less`:
   ```
   ── 2026-04-28T14:32:11Z arte run #b8e3a1 ──────────────────────────
   $ inspect run arte -- 'docker ps --format "{{.Names}}"'
   arte | atlas-vault
   arte | atlas-pg
   arte | aware
   ── exit=0 duration=423ms audit_id=01HXR9Q5YQK2 ──
   ```
   Each block is fenced by `── … ──` lines so `awk '/^── /,/^── exit=/'` can extract individual invocations. The structured audit_id link makes it trivial to round-trip from a transcript hit to the structured audit entry.
3. **What is captured.** Every verb invocation against a namespace: the full argv (after redaction \u2014 see #5), the full local stdout + stderr the operator saw, the final exit code, duration, and the audit_id. F14 script bodies are included by hash + first-10-lines preview (full body is in `~/.inspect/scripts/<sha>.sh`); F15 file transfers record the file paths, sizes, and sha256s; F16 streamed output is captured line-by-line as it arrived; F17 multi-step runs capture each step's block.
4. **Rotation + retention.** Per-day file naturally rotates. A `[history]` config block in `~/.inspect/config.toml` controls retention: `retain_days = 90` (default), `max_total_mb = 500` (cap across all namespaces, oldest-first eviction), `compress_after_days = 7` (gzip files older than this). `inspect history rotate` runs the rotation manually (otherwise it runs lazily on each new-day boundary).
5. **Redaction.** L7 (header / PEM / URL credential redaction) applies to transcript writes the same way it applies to stdout: secret-shaped values are redacted in the transcript file too. Per-namespace `[namespaces.<ns>.history].redact = "strict"|"normal"|"off"` overrides the global default. Default is `"normal"` (matches L7 default). Operators handling exceptionally sensitive workloads can set `disabled = true` per namespace to opt out of transcript capture entirely (audit log is still written).
6. **Management verbs:**
   - `inspect history show [<ns>] [--date YYYY-MM-DD] [--grep <pattern>] [--audit-id <id>]` \u2014 prints transcript blocks matching the filters; default is today's transcript for the most-recently-used namespace. `--audit-id` cross-references back from a structured audit hit.
   - `inspect history list [<ns>]` \u2014 lists transcript files with sizes and date ranges.
   - `inspect history clear <ns> --before YYYY-MM-DD` \u2014 deletes (with confirmation) old transcripts for one namespace.
   - `inspect history --json` for agents.
7. **Performance:** transcript writes are append-only via the same `write-then-rename` pattern as the audit log, batched per verb invocation (one fsync per verb, not per output line). Streaming verbs (F16) buffer transcript writes per-second to avoid fsync amplification. Performance gate: a 10-minute streaming verb produces at most ~60 fsyncs against the transcript file.
8. **Out of scope for v0.1.3:** remote-side transcript ingestion (uploading transcripts to a central store), multi-machine merge, search across multiple operator workstations. Those are v0.1.5 stabilization or v0.2.0 contract decisions. v0.1.3 ships local rotating transcripts only.

**Test:**
- `tests/phase_f_v013.rs::history_captures_invocation_with_audit_link` \u2014 run a sequence of `inspect status`, `inspect run`, `inspect put` against a mock medium; assert the corresponding `~/.inspect/history/<ns>-<today>.log` file exists, contains three fenced blocks in order, each block's `audit_id=` matches the structured audit entry, and the rendered output matches what the operator's stdout received.

- Retention: with `retain_days = 7` and a fixture set of transcripts dated 1\u201310 days ago, `inspect history rotate` deletes the four oldest. Audit log is untouched.
- Compression: with `compress_after_days = 1`, a transcript file dated yesterday is gzipped on next rotation; `inspect history show --date <yesterday>` transparently decompresses on read.
- Redaction: a verb that prints `Authorization: Bearer abc123` to stdout has `Authorization: Bearer ***REDACTED***` (or whatever L7's default rule is) in the transcript; the audit log applies the same rule.
- `disabled = true` per-namespace: no transcript file is written for that namespace; audit log is still written.
- Cross-reference: `inspect history show --audit-id <id>` returns exactly the block whose `audit_id=` matches.

### L4 — Password authentication + extended session TTL + `ssh add-key` helper
**Source:** Roadmap "Remaining work" / known limitation; key-only auth blocks integration with shared bastions and legacy boxes that still require password auth. Refined during v0.1.3 backlog review: password auth is only acceptable if (a) the session TTL is long enough that passwords are not re-prompted within a working session, and (b) there is a one-command path off password auth onto keys.
**Severity:** Medium (blocks adoption on legacy infrastructure).
**Problem:** `inspect connect` only supports key-based SSH auth. Legacy servers, locked-down bastions, and password-only managed appliances cannot be onboarded today. Operators must shell out to plain `ssh` and lose the audit trail / ControlMaster / discovery cache. Even when password auth is available, re-prompting every ~4h within a long working session would make the feature unusable.
**Fix:**
- New per-server config field `auth = "password"` in `~/.inspect/servers.toml`. Default remains `"key"`.
- Password sources, in order: `password_env = "VAR_NAME"` → interactive prompt at `inspect connect`. Never stored on disk.
- **Session TTL.** When `auth = "password"`, the OpenSSH ControlMaster `ControlPersist` defaults to `12h` (was effectively the agent / system default, often 4h). Configurable per-server via `session_ttl = "24h"`. Cap at 24h so a forgotten laptop doesn't hold a live remote session indefinitely; document the cap. Key-auth servers keep the existing default unless explicitly overridden — only password auth gets the longer default, because re-prompting passwords is the painful case.
- Password entered once at `inspect connect`; every subsequent `inspect <verb>` reuses the ControlMaster socket without re-prompting until TTL expires or the user runs `inspect disconnect`.
- **`inspect ssh add-key <ns>` helper** (new verb). Wraps `ssh-copy-id`: generates a key pair under `~/.ssh/inspect_<ns>_ed25519` if none specified, copies the public key to the remote `authorized_keys` over the current (password) session, then offers to flip the server's `auth = "password"` config to `auth = "key"` with `key_path = "..."`. One command from "password-only legacy box" to "first-class key-auth namespace".
  - `--key <path>` to use an existing key instead of generating one.
  - `--no-rewrite-config` to skip the auth-flip prompt (just install the key).
  - Audit-logged like any other write: `verb=ssh.add-key, target=<ns>`.
- One-time warning printed at first password connect: `warning: password auth is less secure than key-based. Run 'inspect ssh add-key <ns>' to migrate.`
- Max 3 failed password attempts then abort with a clear chained hint (point at `inspect help ssh`).
- Update `inspect connectivity` to surface password-auth status, current session TTL, and remaining session lifetime in its triage output.

```toml
[legacy-box]
host = "legacy.internal"
user = "admin"
auth = "password"
password_env = "LEGACY_BOX_PASS"   # optional; falls back to prompt
session_ttl = "12h"                # optional; default 12h for password, capped at 24h
```

**Test:**
- `inspect connect legacy-box` with `LEGACY_BOX_PASS` set succeeds without a prompt; unset, prompts once and connects.
- After connect, `inspect logs legacy-box/<svc>` reuses the session without re-prompting.
- ControlMaster persists for the configured TTL; verify by checking the socket exists 5 minutes after connect and a fresh command does not re-prompt.
- `session_ttl = "48h"` is rejected with a clear error explaining the 24h cap.
- `inspect ssh add-key legacy-box` over a live password session: generates a key, installs it, prompts to flip config; after flip, `inspect connect legacy-box` succeeds with key auth and never prompts.
- 3 wrong passwords aborts cleanly with non-zero exit and a hint.
- `inspect connect` against a `auth = "key"` server is unchanged (regression guard).
- `inspect connectivity legacy-box` shows `auth: password`, `session_ttl: 12h`, `expires_in: 11h47m`.

### L2 — OS keychain integration (opt-in, cross-session only)
**Source:** Roadmap "Remaining work" / known limitation. Refined during v0.1.3 backlog review: keychain is **only useful for cross-session persistence**. Within a single shell session, ssh-agent + ControlMaster already gives "enter passphrase once and reuse for the rest of the session" — that is the default, expected behavior and most users want exactly that (passphrase gone on logout, re-entered next session). Keychain is the opt-in for the smaller group who want passphrases to survive reboot.
**Severity:** Low (opt-in convenience for a subset of users; not on the default path).
**Problem:** A subset of operators want SSH passphrases to persist across shell sessions and reboots without leaving them in env vars, `.envrc` files, or shell history. There is no OS-native option today.
**Default behavior is unchanged and remains the recommended path:** ssh-agent holds the passphrase for the life of the shell session; logout / reboot clears it; next session prompts once again. This is what most users want and the docs should say so explicitly.
**Fix:**
- Add the `keyring` crate (single dep, cross-platform). Backends: macOS Keychain, GNOME Keyring (Secret Service), KDE Wallet, Windows Credential Manager (via WSL2).
- **Opt-in only.** No automatic keychain use. New explicit flag: `inspect connect <ns> --save-passphrase` — prompts once, stores in OS keychain under service `inspect-cli`, account `<namespace>`. Without the flag, behavior is exactly as in v0.1.2 (ssh-agent / per-session prompt).
- Subsequent `inspect connect <ns>` auto-retrieves silently **only if the namespace was previously saved with `--save-passphrase`**.
- New management verbs:
  - `inspect keychain list` — show stored namespaces (no values).
  - `inspect keychain remove <ns>` — delete entry.
  - `inspect keychain test` — verify the OS backend is reachable; exit 0/non-zero with a clear hint when keychain is unavailable.
- New credential-resolution order: socket → user ControlMaster → ssh-agent → **OS keychain (only if `--save-passphrase` was previously used for this ns)** → env var → prompt.
- Headless / CI: skip keychain silently if backend unavailable, fall back to env var or prompt. No hard dep on a running keychain daemon.
- **Inspect never writes passphrases to its own files**, ever. Keychain is the one persistent path, and only when explicitly opted into.
- Docs (`docs/MANUAL.md`) get a short "credential lifetime" section that spells out the three options: (1) **default** — ssh-agent, one prompt per shell session; (2) **`--save-passphrase`** — OS keychain, persists across sessions and reboots; (3) **env var** — for CI / scripted use only.

**Test:**
- Default path (no `--save-passphrase`): `inspect connect arte` behaves identically to v0.1.2 — agent if loaded, otherwise prompt; nothing written to keychain.
- `inspect connect arte --save-passphrase` stores; subsequent connects in a fresh shell don't prompt.
- `inspect keychain list` shows `arte`; `inspect keychain remove arte` deletes; subsequent connect prompts again.
- Headless container with no keychain backend: `inspect connect --save-passphrase` warns once and falls back to per-session prompt; `inspect keychain test` exits non-zero with a hint.
- Order test: a stored keychain entry for `arte` is consulted only because `arte` was previously saved; a fresh ns `bravo` ignores the keychain entirely (no implicit cross-namespace lookups).

### L5 — Audit log rotation + retention policy
**Source:** v0.1.2 bundle implementation retrospective.
**Severity:** Low (no scale issue today, but will bite long-running installations and bundle-heavy workflows).
**Problem:** `~/.inspect/audit/` grows unbounded. Monthly JSONL files are manageable today, but a team running 50 bundle ops per day will accumulate years of entries plus orphaned snapshot directories. There is no rotation, no retention policy, no cleanup verb, and no snapshot GC.
**Fix:**
- New verb `inspect audit gc` with two retention modes:
  - `--keep <duration>` (e.g. `90d`, `4w`, `12h`) — delete audit entries older than N.
  - `--keep <N>` — keep the last N entries per namespace.
  - `--dry-run` — preview what would be deleted (counts + total bytes freed).
- Snapshot directory gets the same treatment in the same pass: any snapshot dir under `~/.inspect/snapshots/` not referenced by a retained audit entry is an orphan and gets cleaned (subject to `--dry-run`).
- New config option in `~/.inspect/config.toml`:
  ```toml
  [audit]
  retention = "90d"     # or "100" for entry count; unset = no automatic GC
  ```
  When set, GC runs lightweight on every `--apply` invocation (just checks the oldest file's mtime; no full scan unless rotation is needed).
- `inspect audit gc --json` for machine consumers (counts, freed bytes, deleted snapshot ids).
- Clear, loud output on what was deleted and what was kept; never silently delete on an unrecognized retention value.

**Test:**
- Backfill `~/.inspect/audit/` with synthetic entries dated >100d in the past; `inspect audit gc --keep 90d --dry-run` lists them; `inspect audit gc --keep 90d` deletes them and the orphaned snapshot dirs.
- `audit.retention = "90d"` in config: an `--apply` invocation triggers GC; second invocation within the same minute does not re-scan (cheap-path guard).
- `--json` output includes `deleted_entries`, `deleted_snapshots`, `freed_bytes`.
- Snapshot referenced by a retained audit entry is **never** deleted.

### L7 — Header / PEM / URL credential redaction in stdout
**Source:** v0.1.2 bundle implementation retrospective.
**Severity:** Medium (security — agent workflows pipe stdout into LLM context windows; one leaked Bearer token can be catastrophic).
**Problem:** The current secret masker (`src/redact.rs`) is line-oriented and only catches `KEY=value` env-var-style strings. It misses:
1. HTTP `Authorization: Bearer <token>` headers in `curl -v` output.
2. PEM private key blocks in file-content dumps (`cat /etc/ssl/private/...`).
3. Credentials embedded in URLs (`postgres://user:pass@host/db`, `redis://:pass@host:6379`).
4. Cookies / `Set-Cookie` headers carrying session tokens.
**Fix:** Add three additional masking patterns running in sequence alongside the existing env-var masker. Each is a separate module under `src/redact/` so future patterns plug in cleanly:

1. **Header masker** (`src/redact/headers.rs`):
   - Match (case-insensitive) `Authorization:`, `X-API-Key:`, `X-Auth-Token:`, `Cookie:`, `Set-Cookie:`, `Proxy-Authorization:`. Mask the value portion: `Authorization: Bearer ****`.
2. **PEM masker** (`src/redact/pem.rs`):
   - Match `-----BEGIN (RSA |EC |DSA |OPENSSH |ENCRYPTED )?PRIVATE KEY-----` through the matching `-----END ... PRIVATE KEY-----`. Replace the entire block (including delimiters) with `[REDACTED PEM PRIVATE KEY (N lines)]`.
3. **URL credential masker** (`src/redact/url_creds.rs`):
   - Match `<scheme>://<user>:<pass>@<host>` in connection strings (postgres, mysql, redis, mongodb, amqp, http, https, etc.). Mask the password portion only: `postgres://user:****@host/db`.

- All maskers run on every `inspect run`, `inspect exec`, `inspect logs`, `inspect search`, `inspect bundle run` stdout/stderr line.
- `--show-secrets` bypasses **all** maskers (extends existing behavior — single flag, single bypass).
- Maskers are streaming-safe (no buffering of full output); PEM masker uses a small line-window state machine.
- Per-masker counter exposed in `--json`: `redactions: { env: 2, header: 1, pem: 0, url: 1 }` so consumers can audit what was masked.

**Test:**
- `inspect run <ns>/<svc> -- 'curl -v https://api.example.com/'` masks the `Authorization:` header.
- `inspect run <ns>/<svc> -- 'cat /etc/ssl/private/test.pem'` replaces the PEM block.
- `inspect run <ns>/<svc> -- 'echo postgres://u:p@h/db'` masks the password.
- `--show-secrets` returns the unmasked output verbatim across all three patterns.
- `--json` output's `redactions` counters match the patterns triggered.
- Streaming test: a slow `curl -v` heartbeat does not break the header mask across chunk boundaries.

### L3 — Parameterized aliases
**Source:** Roadmap "Remaining work" / known limitation; agent + recipe ergonomics.
**Severity:** Medium (parser change + alias storage format change).
**Problem:** Aliases today are static strings. A recipe that wants "logs for *any* service on arte" has to either define one alias per service or fall back to writing the full LogQL each time. Agents can't compose aliases programmatically.
**Fix:** Add `$param` placeholders in alias bodies, supplied at call site as `@name(key=val,key2=val2)`. Aliases may chain other aliases.

```bash
# Define
inspect alias add svc-logs '{server="arte", service="$svc", source="logs"}'

# Use
inspect search '@svc-logs(svc=pulse) |= "error"'

# Chain
inspect alias add prod-pulse '@svc-logs(svc=pulse) |= "$pat"'
inspect search '@prod-pulse(pat=ERROR)'

# Agent discovery
inspect alias show svc-logs --json
# → {"name":"svc-logs","parameters":["svc"],"type":"logql","body":"..."}
```

Design points:
- `$param` syntax in alias body. Tokenizer recognizes `$<ident>` outside of string-literal quoting.
- Call site syntax: `@name(k=v,k=v)`. Bare `@name` still works for parameterless aliases (back-compat with v0.1.2 within the v0.1.x series).
- Missing required param → clear error listing the alias name and required params: `error: alias 'svc-logs' requires param 'svc' (call as @svc-logs(svc=...))`.
- Chain depth cap: 5. Beyond that → error at expansion time with the chain shown.
- Circular reference detection at definition time (`alias add`) — refuse to write the alias and explain the cycle.
- `inspect alias show <name> --json` includes a `parameters: []` list for agent discovery.
- **No defaults in v0.1.3.** `${svc:-pulse}` syntax is deferred to v0.2.0 so the parser stays minimal.
- Storage: `aliases.toml` schema gains an optional `parameters = []` field for cached param list (rebuilt on `alias add`); old aliases without the field continue to work.

**Test:**
- `alias add svc-logs '{service="$svc"}'` then `inspect search '@svc-logs(svc=pulse)'` resolves to `{service="pulse"}`.
- Missing param → exit 2 with the required-param error format.
- Chained aliases work to depth 5; depth 6 errors with the chain printed.
- `alias add a '@b'; alias add b '@a'` — second `alias add` errors with "circular reference: a → b → a".
- `inspect alias show <name> --json` includes the resolved `parameters` array.
- Bare `@name` (no parens) on a parameterless alias is unchanged from v0.1.2.

### L6 — Per-branch rollback tracking in bundle matrix steps
**Source:** v0.1.2 bundle implementation retrospective.
**Severity:** Medium (architectural — affects correctness of parallel rollback in the bundle engine that just shipped).
**Problem:** Bundle matrix steps (`parallel: true` with `matrix:`) currently execute as all-or-nothing per step. If 4 of 6 volume tars succeed and 2 fail, rollback today undoes the entire step — including the 4 successful branches whose output may already have been used downstream. The correct behavior: rollback only the branches that actually succeeded; leave the failed ones alone (nothing to undo).
**Fix:**
- Track per-branch completion status in the bundle executor (`src/bundle/exec.rs`):
  ```rust
  struct BranchResult {
      branch_id: String,        // e.g. "atlas_milvus"
      status: BranchStatus,     // Ok | Failed | Skipped
      audit_id: Option<String>, // per-branch audit entry id
  }
  ```
- On rollback, only execute the rollback block for branches with `status == Ok`. Failed branches are skipped silently (with an audit note explaining why).
- Audit log: each matrix branch gets its own audit entry under the same `bundle_id` and step id, with a `branch:` field for filtering.
- Rollback templating becomes branch-aware: `{{ matrix.<key> }}` inside a `rollback:` block resolves to **only the succeeded branches' values** when the rollback is triggered. (Inside the forward `exec:`, behavior is unchanged — full matrix.)
- New verb `inspect bundle status <bundle_id>` shows per-branch outcomes:
  ```
  bundle: atlas-phase-0-snapshot (id: 0e3a…)
    step: tar-volumes
      ✓ atlas_milvus    (12.3s)
      ✓ atlas_etcd      (4.1s)
      ✗ aware_milvus    (failed: no space left on device)
  ```
- `--json` variant emits the structured per-branch outcomes for agent consumption.
- Integration with B9's existing `on_failure: rollback_to:` — when rolling back **to** a checkpoint that includes a matrix step, all completed branches of that step are kept (rollback target is "everything after this checkpoint"); when rolling back **a** matrix step itself, only succeeded branches are reversed.

**Test:**
- Inject a mid-matrix failure (3 of 5 branches succeed, 2 fail) — rollback runs only for the 3 succeeded branches; audit log shows per-branch entries with correct `status`.
- `inspect bundle status <id>` displays the per-branch outcomes; `--json` matches.
- `rollback:` block referencing `{{ matrix.volume }}` is invoked exactly once per succeeded branch with the correct value substituted.
- A matrix step where all branches succeed and a *later* step fails (triggering rollback to a checkpoint before the matrix): rollback covers all branches as expected (regression guard for the existing path).
- `--json` audit query `inspect audit show --bundle <id>` returns the per-branch entries grouped by step.

### L1 — TUI mode (`inspect tui`)
**Source:** Roadmap "Remaining work" / known limitation; long-standing field request for an interactive dashboard.
**Severity:** Low (large feature, but read-only and additive — does not affect existing surface).
**Problem:** Interactive triage today is a sequence of `inspect status`, `inspect logs --follow`, `inspect why` invocations in separate terminals. Operators want a single live dashboard for at-a-glance health and drill-down. Agents do not need this; humans do.
**Fix:** Ship `inspect tui` — a read-only three-pane dashboard built on `ratatui`. Thin presentation layer over the existing verb functions (no new business logic). Read-only in v0.1.3; write actions inside the TUI are deferred to v0.2.0+.

```
┌─ Services ─────────────────┬─ Logs ──────────────────────────────────┐
│ ▶ pulse        ✓ healthy   │ [pulse]   Request received...           │
│   atlas        ✓ healthy   │ [atlas]   Query OK (42ms)               │
│   synapse      ✗ down      │ [synapse] Connection refused            │
├─────────────────────────────┼─ Detail ────────────────────────────────┤
│                             │ synapse — exited (code 137)             │
│                             │ depends_on: [pulse, atlas, redis]       │
│                             │ Suggested: inspect why arte/synapse     │
└─────────────────────────────┴─────────────────────────────────────────┘
 [q]uit [/]search [f]ollow [m]atch [enter]drill [w]hy [e]xec [?]help
```

Design points:
- **Crate:** `ratatui` + `crossterm`.
- **Left pane:** `inspect status` data, refreshed every 10s (configurable via `--refresh <dur>`).
- **Right-top pane:** `inspect logs --follow --merged` for the selected service(s); supports `/` search and `m` match-filter live.
- **Right-bottom pane:** service detail card (image, ports, depends_on, last health) + suggested `inspect why` invocation; pressing `w` runs `why` and replaces the pane content.
- **Keybindings:** `j`/`k` navigate, `Enter` drill, `/` search, `f` toggle follow, `m` set match filter, `w` run why, `r` force refresh, `?` help overlay, `q` quit. **No mouse.** **No custom layouts.** Fixed three-pane layout.
- **Selector targeting:** `inspect tui [<namespace>|<selector>]` — defaults to all configured namespaces; narrows to a namespace or selector when given.
- **Read-only:** any keypress that would cause a write action (`e`, `x`, etc. in v0.2.0+) shows `read-only in v0.1.3` in the status bar and exits the keypress.
- **Crash safety:** restore terminal state on panic via `crossterm::terminal::disable_raw_mode` in a panic hook; never leave the user's terminal in a broken state.
- **CI:** smoke test runs `inspect tui --selftest` which initializes the layout, ticks the refresh once, dumps the rendered frame to stdout as text, and exits 0.
- **Scope discipline:** no themes, no plugins, no custom layouts, no per-cell styling configuration in v0.1.3. The TUI is a window onto existing commands, nothing more.

**Test:**
- `inspect tui --selftest` exits 0 within 2s and prints a layout snapshot to stdout.
- Snapshot test: rendered frame for a fixed mock dataset matches a golden file in `tests/golden/tui/`.
- Panic hook test: forcibly panic inside the render loop, verify terminal mode is restored (`stty -a` snapshot returns to cooked mode).
- Keybinding test: each documented key produces the documented effect against a mock data backend (no real SSH).
- `inspect tui` against an unreachable namespace shows the namespace as `unreachable` in the left pane and surfaces the underlying connectivity error in the detail pane (no crash).
